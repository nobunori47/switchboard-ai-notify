# Case 5: SwitchBoard — AI Multi-Channel Notification Hub

## 背景

不動産管理会社では、メール・LINE・電話に問い合わせが分散し、
特に営業時間外のLINE問い合わせが見落とされやすいという課題があった。
また、クレームのような緊急性の高い問い合わせほど対応が遅れ、
信頼低下につながっていた。

## 課題

- 月間750件規模の問い合わせが複数チャネルに分散し、管理が煩雑
- 振り分け作業に時間がかかり、初動対応が遅れる
- クレーム対応の遅延がそのまま顧客満足度の低下に直結する
- 初期費用15万円・月額1〜2万円という低予算での実現が前提

## 設計判断

技術選定より先に、課題と設計の対応関係を明確にすることを優先した。

| 課題 | 設計 | 技術 |
|---|---|---|
| 問い合わせの分散 | Slackへの自動集約 | Webhook + Slack Web API |
| 緊急対応の遅延 | 通常パスと分離した即時通知経路 | Webhook受信と同一関数内で完結する緊急パス |
| 振り分け作業の負荷 | AIによる4カテゴリ自動分類 | Claude API + Headless Cron |
| 低予算・低運用コスト | 常時稼働サーバーを持たない構成 | Vercel Functions + Supabase |

## 工夫した点

**冪等性(二重通知防止)**
LINE・GmailのWebhookは再送されることがあるため、`external_id`にUNIQUE制約を設け、
同一メッセージの二重処理をデータベース側で防ぐ設計にした。

**緊急パスの二段階判定**
「至急」「クレーム」等のキーワードでまず一次判定し、次にClaudeで文脈を再確認する
二段階構成にすることで、「至急日程を決めたい(緊急ではない)」のような
キーワードだけでは誤検知しやすいケースにも対応した。

**状態遷移の一貫性**
通知処理は「実際にSlack/LINEへの送信が成功した後」にのみ`notified`へ進める設計にした。
分類が完了した時点で状態を進めてしまうと、送信が失敗していても
「届いたはず」に見えてしまうという、通知システムで最も避けたい事故につながるため。

**AI判断の可視化**
分類結果だけでなく、信頼度スコア(`confidence`)と判断理由(`classification_reason`)を
保存し、AIの判断をブラックボックス化しない設計にした。

## 技術構成

Next.js(App Router)/ TypeScript / Supabase / Claude API / Slack Web API /
LINE Messaging API / Vercel Cron(Headlessモード)
